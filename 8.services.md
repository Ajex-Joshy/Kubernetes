## Services

since the pods are temporary (ephemeral) their IP addresses keep changing, so kubernetes uses a service to give fixed IP or DNS name to access the pods

A service is a kubernetes object that provides a stable network endpoint to access a group of pods

if the clients are directly connected to pods , connection breaks when the pod restarts because the IP address of new pod is different but services keeps the same IP and DNS name. So. clients interact with services

But how does services communicate with pods?. Do they keep track of IP addresses?

The answer is no. A service does not directly know pod IPs manually.
Services uses labels and selectors to identify pods.

Now the services found the Pods, but to communicate with these pods we need to know the IPs of this pods.

so for that kubernetes creates an Endponts object.
Endpoints are live list of pod IPs that match the service selector
If a pod is deleted or added the IPs are automatically updated to this endpoints.

now we have service for the client, and Endpoints store the list of pod IP addresses that the service should send traffic to. but how does service decide which pod should receive the request?

This is handled by kube-proxy

Every kuberentes node runs a component called kube-proxy.
kube-proxy watches

- services
- endpoints

when endpoints are created or updated kube-proxy creates network routing rules inside the node.

So when a request comes to the service IP, kubeproxy forwards the request to one of the pod Ips listed in the endpoints.
It routes traffic and distributes it across pods.

so in short
**A Service provides a stable entry point, Endpoints maintain the list of pod IPs, and kube-proxy routes traffic from the service to one of the pods.**

    •	Clients talk to the Service
    •	Service uses Endpoints to know Pod IPs
    •	kube-proxy routes the traffic

When you create a service, kubernetes assigns it an IP called ClusterIP
This IP belongs to the cluster internal network

so when another pod calls http://backend-service
kubernetes DNS resolves it to - 10.96.0.10

As I said earlier only things inside the cluster can access clusterIp

eg -

```
Frontend Pod → backend-service
Worker Pod → backend-service
Another microservice → backend-service
```

So now a question arises how do external users access the app?

kubernetes provides different service types

```
ClusterIP → Internal communication
NodePort → Expose service on node IP
LoadBalancer → Cloud load balancer
ExternalName →  Maps a service to an external DNS name
```

so suppose we have to expose a service to public we can expose a port in node IP address
lets say the IP address of node is 192.168.1.10
we can expose the service in port of range - 30000-32767, lets say we chooses 30007
so the public address will be: http://192.168.1.10:30007
This is NodePort service type

but in production systems, traffic usually enters the cluster through an Ingress controller.
which we will look into detail later.
