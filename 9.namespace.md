# Kubernetes Namespaces

## What is a Namespace?

A **Namespace** in Kubernetes is a way to logically divide a single Kubernetes cluster into multiple virtual environments. It allows different teams, applications, or environments to share the same cluster while keeping their resources separated and organized.

Namespaces help manage and isolate resources such as:

- Pods
- Services
- Deployments
- ConfigMaps
- Secrets
- StatefulSets
- Jobs

Think of namespaces like **folders inside a computer**. Each folder can contain files with the same names without conflicts.

Example analogy:

```
Computer
 ├── Folder: Projects
 │     └── app.txt
 └── Folder: Notes
       └── app.txt
```

Both folders can contain a file named `app.txt` because they are separated.

Kubernetes works similarly.

---

# Why Namespaces Are Needed

In a real Kubernetes cluster, multiple applications and teams may run workloads.

Without namespaces, all resources would exist in a single global space:

```
Cluster
 ├── nginx
 ├── backend
 ├── redis
 ├── monitoring
 └── logging
```

Problems with this approach:

- Resource name conflicts
- Difficult resource management
- Hard to apply access control
- Hard to separate environments like dev, staging, and production

Namespaces solve this problem by creating logical isolation.

Example:

```
Cluster
 ├── Namespace: dev
 │     ├── frontend
 │     └── backend
 │
 ├── Namespace: staging
 │     ├── frontend
 │     └── backend
 │
 └── Namespace: prod
       ├── frontend
       └── backend
```

Each namespace can contain resources with the **same names** without conflict.

# Default Namespaces in Kubernetes

When a Kubernetes cluster is created, several namespaces already exist.

You can see them using:

```
kubectl get namespaces
```

Example output:

```
default
kube-system
kube-public
kube-node-lease
```

Explanation:

### default

If you do not specify a namespace when creating resources, Kubernetes places them in the **default namespace**.

### kube-system

Contains internal Kubernetes components such as:

- kube-dns / CoreDNS
- kube-proxy
- scheduler
- controller manager

These are critical system services.

### kube-public

Contains resources that should be publicly readable by all users.

### kube-node-lease

Used for node heartbeat information to help Kubernetes detect node failures quickly.

---

# Creating a Namespace

You can create a namespace using the CLI.

```
kubectl create namespace dev
```

Verify:

```
kubectl get namespaces
```

---

# Creating Resources Inside a Namespace

Resources can be created inside a namespace in two ways.

## 1. Using CLI

```
kubectl run nginx --image=nginx -n dev
```

Here:

```
-n dev
```

specifies the namespace.

---

## 2. Using YAML Configuration

Example:

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: nginx
  namespace: dev

spec:
  containers:
    - name: nginx
      image: nginx
```

Apply:

```
kubectl apply -f pod.yaml
```

---

# Viewing Resources in a Namespace

To see resources inside a specific namespace:

```
kubectl get pods -n dev
```

or

```
kubectl get services -n dev
```

If you do not specify `-n`, Kubernetes assumes the **default namespace**.

---

# Setting Default Namespace for Current Context

Instead of specifying `-n` every time, you can change the default namespace.

```
kubectl config set-context --current --namespace=dev
```

Now all commands will use the `dev` namespace unless specified otherwise.

Example:

```
kubectl get pods
```

will now show pods in the **dev namespace**.

---

# Namespace Isolation

Namespaces provide logical isolation but **not full security isolation**.

Pods in different namespaces can still communicate unless restricted using:

- Network Policies
- RBAC (Role-Based Access Control)
- Resource Quotas

Example communication:

```
frontend.dev.svc.cluster.local
backend.prod.svc.cluster.local
```

---

# Kubernetes DNS with Namespaces

Services are automatically registered in Kubernetes DNS.

Service naming format:

```
service-name.namespace.svc.cluster.local
```

Example:

```
backend.dev.svc.cluster.local
```

Inside the same namespace you can simply call:

```
http://backend
```

---

# Best Practices for Namespaces

Common namespace structure in production clusters:

```
Cluster
 ├── dev
 ├── staging
 ├── production
 ├── monitoring
 ├── logging
 └── ingress
```

Benefits:

- environment isolation
- easier resource management
- better access control
- easier debugging

---

# Summary

Namespaces are a fundamental Kubernetes concept used for organizing and isolating resources within a cluster.

Key points:

- Namespace = logical partition inside a cluster
- Allows multiple teams/applications to share the same cluster
- Prevents resource naming conflicts
- Helps apply security and resource limits
- Enables environment separation (dev, staging, prod)
