## Deployment

A **Deployment** is a higher-level controller in Kubernetes that manages **ReplicaSets**, which in turn manage **Pods**.

It is the most common way to run applications in production Kubernetes clusters because it provides features like **rolling updates, rollbacks, and scaling**.

### Deployment Architecture

```
Deployment
    │
    ▼
ReplicaSet
    │
    ▼
Pods
```

- **Deployment** – Manages application updates and desired state
- **ReplicaSet** – Ensures a specified number of pods are running
- **Pods** – Run the actual containers

---

## Why Deployments are Used

While **ReplicaSets** can maintain a fixed number of pods, they cannot handle application lifecycle tasks such as:

- Updating application versions
- Performing rolling updates
- Rolling back to a previous version
- Managing release history

Deployments solve these problems.

Example production workflow:

```
Version v1 running
       ↓
Deploy v2
       ↓
Rolling update replaces old pods gradually
       ↓
If failure occurs → Rollback to v1
```

---

## Key Responsibilities of a Deployment

A Deployment controller is responsible for:

1. Creating and managing ReplicaSets
2. Updating pods when application versions change
3. Scaling pods up or down
4. Maintaining application availability during updates
5. Allowing rollbacks to previous versions

---

## How Deployments Work

When you create a Deployment:

1. Kubernetes creates a **ReplicaSet**
2. The ReplicaSet creates **Pods**
3. The Deployment continuously monitors the system
4. If the state changes, Kubernetes automatically corrects it

Example:

Desired state

```
replicas: 3
```

If one pod crashes:

```
Pod deleted/crashed
        ↓
ReplicaSet detects missing pod
        ↓
New pod created automatically
```

---

## Deployment vs ReplicaSet

| Feature            | ReplicaSet | Deployment  |
| ------------------ | ---------- | ----------- |
| Maintain pod count | Yes        | Yes         |
| Rolling updates    | No         | Yes         |
| Rollback support   | No         | Yes         |
| Version history    | No         | Yes         |
| Production usage   | Rare       | Very common |

In production systems, developers **rarely create ReplicaSets directly**. Instead, they create **Deployments**, which automatically manage ReplicaSets.

---

## Important Note

**Do not manually modify ReplicaSets that are owned by a Deployment.**

Because:

- The Deployment controller continuously monitors ReplicaSets.
- If you manually change them, the Deployment may **override your changes**.

Instead, always modify the **Deployment configuration**, and Kubernetes will update the underlying ReplicaSets automatically.
