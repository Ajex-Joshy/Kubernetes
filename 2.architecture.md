# Kubernetes Architecture

A Kubernetes cluster consists of two main parts:

1. **Control Plane** – Manages and controls the cluster
2. **Worker Nodes** – Machines where applications run

---

## Control Plane

The **Control Plane** manages the overall state of the cluster.

It consists of the following components:

### 1. API Server (kube-apiserver)

- Entry point of the Kubernetes cluster
- All requests from CLI, UI, or APIs go through the API server
- Validates and processes requests
- Communicates with other control plane components

Example:

```bash
kubectl get pods
```

---

### 2. Scheduler (kube-scheduler)

Responsible for **assigning pods to nodes**.

The scheduler decides which node should run a pod based on:

- CPU availability
- Memory availability
- Resource requirements
- Node constraints

> The scheduler **only assigns pods to nodes**, it does not create them.

---

### 3. etcd

A **distributed key-value database** used to store cluster data.

It stores:

- Cluster configuration
- Pod information
- Node information
- Secrets
- ConfigMaps
- Deployments

Think of **etcd as the database of the cluster**.

---

### 4. Controller Manager (kube-controller-manager)

Runs controllers that maintain the **desired state** of the cluster.

Examples of controllers:

| Controller             | Responsibility            |
| ---------------------- | ------------------------- |
| Node Controller        | Monitors node health      |
| Replication Controller | Maintains pod replicas    |
| Endpoint Controller    | Updates service endpoints |
| Job Controller         | Manages batch jobs        |

Example:

```
replicas: 3
```

If only **2 pods are running**, the controller automatically **creates another pod**.

---

## Worker Nodes

Worker nodes are the machines where containers actually run.

Each worker node contains the following components:

### 1. kubelet

Agent running on every worker node.

Responsibilities:

- Communicates with the API server
- Ensures containers described in Pod specifications are running
- Monitors pod health

Flow:

```
API Server → kubelet → run containers
```

---

### 2. kube-proxy

Handles **network communication inside the cluster**.

Responsibilities:

- Maintains network rules
- Enables Pod-to-Pod communication
- Enables service load balancing

Example:

```
Service → forwards traffic → correct Pod
```

---

### 3. Pods

A **Pod** is the **smallest deployable unit in Kubernetes**.

A pod can contain:

- One or more containers
- Shared network
- Shared storage
- A single IP address

Example:

```
Pod
 ├── Container (Node.js)
 └── Container (Redis sidecar)
```

---

### 4. Container Runtime

Software responsible for running containers.

Examples:

- containerd
- CRI-O
- Docker (earlier widely used)

Responsibilities:

- Pull container images
- Create containers
- Run containers

---

## kubectl

kubectl is the primary command-line tool for communicating with a Kubernetes cluster
Examples:

```bash
kubectl get pods
kubectl get nodes
kubectl apply -f deployment.yaml
```

Flow:

```
kubectl → API Server → Cluster
```

The kubectl tool supports many operations, which fall into these broad categories:

- **Manage resources** – Create, update, and delete objects such as Pods, Deployments, and Services. Use kubectl apply for declarative management from configuration files.
- **Inspect cluster state** – List and describe objects, view events, and check resource usage.
- **Debug** – View logs from containers, execute commands inside a running container, or port-forward to a Pod.
- **Cluster operations** – Drain nodes for maintenance, cordon nodes to prevent new workloads, and manage cluster configuration.
  Script and automate – Format output as JSON, YAML, or custom columns using JSONPath for use in scripts and pipelines.

---

## Local Kubernetes Tools

### Minikube

Used to run a **single-node Kubernetes cluster locally**.

Good for:

- Learning Kubernetes
- Local testing

---

### Kind

Kind stands for **Kubernetes IN Docker**.

Used to:

- Run Kubernetes clusters using Docker containers
- Test Kubernetes setups locally

---

## Deployment Flow

When deploying an application:

```
kubectl apply -f deployment.yaml
        ↓
API Server receives request
        ↓
etcd stores cluster configuration
        ↓
Scheduler selects a node
        ↓
kubelet runs the pod on the worker node
        ↓
Container runtime starts the container
```
